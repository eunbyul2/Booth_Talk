# Booth Talk 코드베이스 평가 및 피봇 감사 보고서

## 1. 개요

본 문서는 'Booth Talk' 서비스의 코드베이스를 기반으로 '한국 정신건강&심리학 컨퍼런스/세미나' 정보 제공 사이트로의 비즈니스 피봇 가능성을 평가하고, 이를 위한 코드 감사 및 실행 계획을 제시합니다.

**결론부터 말하자면, Booth Talk의 코드베이스는 제안된 피봇을 위한 매우 견고하고 훌륭한 기반입니다.** 기존 코드는 이벤트 기반 서비스의 핵심 기능을 이미 갖추고 있어, 완전히 새로 개발하는 대신 **리팩토링 및 확장**을 통해 효율적으로 목표를 달성할 수 있습니다.

## 2. 코드베이스 평가

### 가. 기술 스택 (Technology Stack)

- **Backend:** Python, FastAPI
- **Frontend:** JavaScript, React
- **Database:** PostgreSQL
- **Crawling:** Python (requests, BeautifulSoup)

최신 기술 스택을 사용하여 유지보수가 용이하고 확장에 유연한 구조입니다.

### 나. 아키텍처 (Architecture)

- **모듈형 구조:** 백엔드(`routes`, `services`, `models`)와 프론트엔드(`pages`, `components`) 모두 기능/역할에 따라 명확하게 분리되어 있습니다. 이는 코드의 이해도를 높이고 특정 기능을 수정하거나 제거하기 용이하게 만듭니다.
- **역할 기반 분리:** `방문자(visitor)`, `기업(company)`, `관리자(admin)` 역할에 따라 API 엔드포인트와 프론트엔드 페이지가 명확히 구분되어 있어, 피봇 과정에서 불필요한 부분을 식별하고 제거하기에 매우 유리합니다.

### 다. 재사용성 평가

- **매우 높은 재사용성:** 이벤트 목록, 상세 정보, 필터링 등 새로운 서비스의 핵심 기능 대부분이 이미 구현되어 있습니다.
  - **크롤러 (`database/koea_crawler.py`, `setec_crawler.py`):** 새 비즈니스 모델의 가장 중요한 부분인 데이터 수집(크롤링)에 대한 **검증된 패턴**을 제공합니다. 이는 피봇 과정에서 가장 큰 자산이며, 개발 시간을 획기적으로 단축시켜 줄 것입니다.
  - **백엔드 (`backend/routes/events_visitor.py`, `backend/models/event.py`):** 방문자를 위한 이벤트 조회 API와 데이터 모델은 거의 그대로 재사용할 수 있습니다.
  - **프론트엔드 (`frontend/src/pages/visitor/EventList.jsx`):** 이벤트 목록을 표시하는 React 컴포넌트는 약간의 UI 수정만으로 즉시 재활용 가능합니다.
  - **데이터베이스 (`database/schema.sql`):** `events`, `venues`, `tags` 테이블 스키마는 새로운 '컨퍼런스/세미나' 정보를 저장하는 데 완벽하게 부합합니다.

## 3. 코드 감사 및 피봇 실행 계획

피봇의 핵심은 **'Booth Talk'의 B2B 관련 기능을 제거하고, 크롤링 대상을 변경하며, UI를 리브랜딩**하는 것입니다.

### 1단계: 불필요한 기능 제거 (Pruning)

새로운 서비스에 필요 없는 기존 'Booth Talk'의 기업 및 설문조사 관련 기능을 과감히 제거하여 코드베이스를 단순화합니다.

- **제거 대상:**
  - **Backend:** `companies`, `surveys`, `event_managers`와 관련된 모든 라우트, 서비스, 모델
  - **Database:** `companies`, `surveys`, `survey_responses`, `event_managers` 테이블
  - **Frontend:** `pages/company/`, `pages/admin/` 하위의 기업 계정 및 설문조사 관련 페이지/컴포넌트

### 2단계: 신규 크롤러 개발 (Development)

기존 크롤러(`koea_crawler.py`, `setec_crawler.py`)를 **템플릿**으로 삼아, 정신건강 및 심리학 관련 학회/기관 웹사이트를 크롤링하는 새로운 파이썬 스크립트를 개발합니다. 이는 피봇의 핵심 개발 작업이 될 것입니다.

### 3단계: UI/UX 리브랜딩 (Rebranding)

- **디자인 변경:** 새로운 서비스의 정체성에 맞게 로고, 색상, 폰트 등 전체적인 CSS 스타일을 수정합니다.
- **용어 변경:** 프론트엔드 코드 내의 '이벤트', '부스' 등의 용어를 '컨퍼런스', '세미나', '발표' 등 새로운 도메인에 맞는 용어로 변경합니다.

## 4. 결론

이 프로젝트는 '재개발'이 아닌 '리팩토링 및 확장'에 가깝습니다. 기존 코드베이스는 이미 새로운 서비스의 가장 복잡한 부분(이벤트 기반 구조, 데이터 크롤링 패턴)을 해결해 놓았기 때문에, 매우 효율적으로 비즈니스 피봇을 진행할 수 있습니다.

## 5. 주요 관련 파일

- `backend/main.py`: FastAPI 백엔드의 전체 구조를 파악할 수 있는 핵심 파일.
- `database/schema.sql`: 재사용할 테이블과 제거할 테이블을 명확히 보여주는 데이터베이스 설계도.
- `database/koea_crawler.py`, `setec_crawler.py`: **새로운 크롤러를 개발하기 위한 가장 중요한 참고 자료이자 템플릿.**
- `frontend/src/pages/visitor/EventList.jsx`: 재사용 가능한 핵심 프론트엔드 컴포넌트.
- `backend/routes/events_visitor.py`: 재사용 가능한 핵심 백엔드 API.
- `backend/models/event.py`: 재사용 가능한 핵심 데이터 모델.

## 6. 상용 서비스 적용 가능성 평가

프로토타입 수준을 넘어 실제 사용자가 있는 상용 서비스를 목표로 할 경우, 현재 코드베이스의 강점을 살리면서 몇 가지 중요한 사항을 보완해야 합니다.

### 가. 상용화를 위한 강점

- **확장성 있는 기술 스택:** FastAPI와 React는 현대적인 웹 서비스의 트래픽을 감당하기에 충분히 효율적이고 확장성이 뛰어납니다.
- **유지보수가 용이한 아키텍처:** 잘 분리된 모듈형 구조는 기능 추가 및 수정 시 발생할 수 있는 오류를 줄여주고, 안정적인 서비스 운영에 기여합니다.
- **검증된 핵심 로직:** 이벤트/컨퍼런스 정보를 수집하고 사용자에게 보여주는 핵심 비즈니스 로직이 이미 구현되고 검증되었습니다.

### 나. 상용화를 위한 보완 사항

1.  **크롤러 고도화:**
    - **안정성:** 특정 웹사이트의 구조 변경, 네트워크 오류 등 다양한 예외 상황을 처리할 수 있는 강력한 오류 처리 및 재시도 로직이 필요합니다.
    - **자동화:** 크롤러가 주기적으로 자동 실행되도록 스케줄링 기능(예: Cron, Celery Beat)을 구현해야 합니다.
    - **모니터링:** 크롤링 성공/실패 여부, 수집된 데이터의 양 등을 기록하고 확인할 수 있는 로깅 및 알림 기능이 필요합니다.

2.  **테스트 코드 도입:**
    - 서비스의 안정성과 신뢰도를 보장하기 위해 **단위 테스트 및 통합 테스트** 코드 작성이 필수적입니다.
    - **Backend:** `pytest`를 사용하여 API 엔드포인트, 서비스 로직 등을 테스트합니다.
    - **Frontend:** `Jest`, `React Testing Library` 등을 사용하여 컴포넌트와 사용자 상호작용을 테스트합니다.

3.  **보안 강화:**
    - **인증/인가:** 관리자 기능이 남아있다면, `Auth.py`의 인증 로직을 강화하고 역할 기반 접근 제어를 명확히 해야 합니다.
    - **입력값 검증:** SQL Injection, XSS(Cross-Site Scripting) 등의 웹 취약점을 방지하기 위해 모든 사용자 입력값에 대한 철저한 검증이 필요합니다.

4.  **배포 및 운영 (DevOps):**
    - **컨테이너화:** `Docker`를 사용하여 애플리케이션을 컨테이너화하면, 개발 및 운영 환경을 일치시키고 배포를 단순화할 수 있습니다.
    - **CI/CD 파이프라인:** GitHub Actions, Jenkins 등을 사용하여 테스트 및 배포 과정을 자동화하면, 빠르고 안정적인 업데이트가 가능해집니다.
    - **환경 변수 관리:** 데이터베이스 접속 정보, API 키 등 민감한 정보를 코드에서 분리하여 `.env` 파일을 통해 안전하게 관리해야 합니다. (`.env.example` 파일이 이미 좋은 출발점입니다.)

5.  **로깅 및 모니터링:**
    - 애플리케이션 전반의 상태, 오류, 사용자 요청 등을 추적할 수 있는 **구조화된 로깅(Structured Logging)** 시스템을 구축해야 합니다.
    - Datadog, Sentry와 같은 외부 모니터링 도구를 연동하여 실시간으로 서비스 상태를 확인하고 문제가 발생했을 때 즉시 대응할 수 있도록 준비해야 합니다.

### 다. 최종 의견

현재 코드베이스는 상용 서비스를 위한 **훌륭한 출발점**입니다. 위에 제시된 '보완 사항'들을 체계적으로 적용해 나간다면, 기술적으로 충분히 안정적이고 확장 가능한 상용 서비스를 성공적으로 구축할 수 있을 것입니다.
